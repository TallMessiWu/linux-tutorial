01如何用这份教程

先约定几个贯穿全文的符号和自查习惯,后面就不再重复解释了。

提示符:$ 和 # 的区别

你在终端里看到的行首符号叫提示符(prompt)。本教程的命令示例都以它开头, 但你复制时不要把它一起敲进去。两种最常见:

符号含义典型完整提示符
$普通用户user@server:~$
#root(超级用户),权力最大,危险操作前务必看清root@server:~#
📖 阅读约定

本文代码块里:绿色 $ 是提示符(别敲); # 灰色斜体 是注释(给你看的说明,不是命令); 暗灰文字是命令"跑出来的输出"。

命令的通用结构

命令+ 选项 -a --all+ 参数(对象) 
$ ls -l -a /etc      # -l=长格式(每行一个,显示权限/大小/时间) -a=连隐藏文件也列出
$ ls -la /etc         # 短选项可合并:-la 等价于 -l -a
$ ls --all /etc        # --all 是 -a 的长写法;长选项用两个减号,通常更易读

(每个选项具体干嘛,第 2 章有完整的 ls 参数表;这里先看懂"命令 + 选项 + 对象"这个结构即可。)

  • 短选项:单减号 + 单字母,可合并,如 -la。
  • 长选项:双减号 + 单词,如 --all;带值时写 --width=80 或 --width 80。
  • -- 单独出现表示"选项到此为止",后面即使以 - 开头也当普通参数(用来处理名字叫 -foo 的文件)。

不会用就自查:三件套

方式用途例子
命令 --help快速看选项清单,最常用tar --help
man 命令完整手册(用 / 搜索、q 退出)man grep
tldr 命令只给常用范例(需另装),比 man 友好tldr tar
💡 在 man 里怎么动

翻页 Space / b,搜索输入 / 再打关键词回车,n 跳下一个匹配,q 退出。 这套快捷键和后面 Vim、less 是相通的——学一次,处处能用。

⚠️ 三条保命习惯(贯穿全文)

① rm 没有回收站,删了就没了,尤其 rm -rf; ② > 会清空目标文件,想追加用 >>(详见第 3 章); ③ 危险命令先把 rm 换成 ls/echo 跑一遍,确认匹配到的是你想要的东西,再真删。

你已经会 cd/ls,这里把"路径到底怎么写"和那些你一直没搞清的参数补齐。

路径:绝对 vs 相对,以及几个特殊符号

写法含义

参数作用

drwxr-xr-x 12 user user 4.0K Jun  6 22:11 ..

tree 不是默认自带,提示 command not found 时安装: Debian/Ubuntu sudo apt install tree、CentOS/RHEL sudo yum install tree、Mac brew install tree。

常用参数

参数作用
-L 2只展开 2 层(Level)。目录深时最有用,否则刷屏
-d只显示目录,不列文件(看骨架)
-a连隐藏文件(. 开头)也显示
-h显示每个文件大小(human readable)
-f每项显示完整相对路径(方便复制)
-I "模式"排除匹配的目录/文件,如 -I "node_modules|.git"
--gitignore按 .gitignore 规则自动忽略(新版本支持)
 
$ tree -L 2                       # 只看前两层,摸清项目骨架最常用
$ tree -d -L 2                    # 只看目录、两层:一眼看清模块划分
$ tree -I "node_modules|.git|__pycache__"  # 排除噪音目录(否则会爆屏)
$ tree -ah | less                 # 太长就接 less 分页翻(见上表 less 用法)
⚠️ 在大项目根目录别直接裸跑 tree

node_modules、.git 这类目录文件成千上万,直接 tree 会刷屏好几分钟。 养成习惯:加 -L 2 限制深度,或用 -I 排除掉它们。

💡 顺手用法

把结构贴进文档/issue:tree -L 2 > structure.txt(重定向见第 3 章)。 tree 默认就给目录加颜色,管道到文件时颜色会自动去掉,无需额外处理。

通配符(glob):* ? [ ]

这是 shell 在执行前帮你把文件名补全展开的机制,注意它和第 4 章的"正则"不是一回事(最常见的混淆点)。

通配符匹配例子
*任意多个任意字符(含 0 个)ls *.log → 所有 .log 文件
?恰好一个任意字符ls log?.txt → log1.txt、logA.txt
[abc]方括号里任意一个字符ls log[12].txt → log1/log2
[0-9]范围ls 2024-[0-1][0-9].csv
{a,b}大括号展开(逐项生成,非"匹配现有文件")cp a.{txt,bak}
⚠️ glob 和正则的核心区别

在 glob 里 * = "任意多字符";但在正则(grep)里 * = "前一个字符重复任意次",含义完全不同! 而且 glob 是谁来展开?是 shell——所以 ls *.log 其实是 shell 先把 *.log 换成真实文件名列表,再交给 ls。这也解释了:没有匹配文件时, *.log 会原样传给命令,可能报"找不到 *.log"。

03管道与重定向 ★

你说"| > >> 一知半解"——这一章就把它们彻底讲透。先建立一个心智模型。

关键模型:每个命令都有 3 根"管子"

任何命令运行时,系统都给它接了三根数据通道(就是所谓文件描述符,记住编号即可,底层细节不必深究):

编号名字默认接到哪装什么
0stdin(标准输入)你的键盘命令的输入
1stdout(标准输出)屏幕正常结果
2stderr(标准错误)屏幕报错信息
📖 为什么要分 1 和 2?

正常结果和报错分两根管子走,你才能"只保存结果、错误另存",或"把结果传给下一个命令、错误仍打屏幕"。 这正是 2>、2>&1 存在的意义。屏幕上你同时看到两者,但它们其实是分开的。

重定向:把管子改接到文件

符号作用记忆
> file把 stdout 写入文件,先清空再写覆盖
>> file把 stdout 追加到文件末尾追加
< file把文件内容喂给命令的 stdin读入
2> file只把错误写入文件错误另存
2>&1把错误"并入"标准输出那根管子错误跟着结果走
&> file结果 + 错误一起写入文件(等于 >file 2>&1)全都存
/dev/null"黑洞"文件,丢进去的内容直接消失扔掉
 
$ echo "第一行" >  log.txt   # 写入(原内容被清空)
$ echo "第二行" >> log.txt   # 追加,现在有两行
$ cat log.txt
第一行
第二行
⚠️ 最常见的事故:> 会清空文件

command > file 会在命令运行前就把 file 截断成 0 字节。 所以 sort file > file 这种"原地处理"会直接清空 file(还没读就被清了)! 要原地改,得用临时文件或 sed -i。想保留旧内容,永远记得是 >> 不是 >。

把错误处理明白:2> 和 2>&1

$ find / -name id_rsa 2> /dev/null   # 结果打屏,"Permission denied"等错误丢黑洞
$ make > build.log 2>&1                 # 结果和错误全写进 build.log
$ ./run.sh > out.log 2> err.log        # 结果、错误分开两个文件
⚠️ 2>&1 的顺序陷阱

2>&1 的意思是"让 2 号管子,接到 1 号此刻所指的地方"。所以顺序很重要:

cmd >file 2>&1 ✅ → 先把 1 指向 file,再让 2 也指向 file(都进文件)。
cmd 2>&1 >file ❌ → 先让 2 指向"当前的 1=屏幕",之后才把 1 改到 file。结果:错误还在屏幕,只有正常输出进了文件。

管道 |:把上一个的 stdout 接到下一个的 stdin

这是 Linux 的灵魂。| 不经过磁盘文件,直接把左边命令的标准输出,当作右边命令的标准输入。

ls -l │ stdout→stdin grep .log │ wc -l → 结果 
$ ls -l | grep '.log' | wc -l
3                                  # 列目录 → 只留含 .log 的行 → 数行数 = 有 3 个
📖 管道只传 stdout(1 号)

错误(2 号)默认不进管道,会直接打在屏幕上。想让错误也一起流过管道,要写 cmd 2>&1 | next(把 2 并入 1,管道才带得走它)。

三个高频配角:tee、xargs、<<<

tee —— 一边存文件,一边继续往下传

名字来自水管的 T 形三通:数据流到这里分两路。常用于"既要看/继续处理,又要留存"。

$ make 2>&1 | tee build.log | grep error
# 完整输出存进 build.log,同时只把含 error 的行显示在屏幕
$ echo "hi" | sudo tee -a /etc/hosts   # 用 sudo 往受保护文件追加(-a=追加)
💡 为什么 sudo echo "x" >> /etc/file 不行?

因为重定向 >> 是 shell 做的,而 shell 是普通权限,sudo 只提升了 echo。 正确做法就是上面的 ... | sudo tee -a 文件——让有 root 权限的 tee 去写。

xargs —— 把"一串文本"变成"命令的参数"

有些命令(如 rm、kill)只认参数,不读 stdin。xargs 负责把管道里的文本,拼成它们的参数。

$ find . -name '*.tmp' | xargs rm          # 把找到的文件名拼给 rm 删掉
$ find . -name '*.tmp' -print0 | xargs -0 rm  # 文件名有空格时的安全写法
⚠️ 文件名带空格会出事

默认 xargs 按空格/换行切分,文件名 my file.tmp 会被当成两个参数。 配对使用 find -print0 和 xargs -0(用不可见的 NUL 字符分隔),就万无一失。详见第 14 章。

这些都是"喂输入"的近亲

写法含义
cmd < file把文件当输入
cmd << EOF ... EOFhere-doc:多行文本直接喂给命令(写配置/邮件正文常用)
cmd <<< "字符串"here-string:把一个字符串当输入,如 grep foo <<< "$var"

04grep 与正则 ★

你说"知道 grep 能抓匹配项,但不清楚怎么匹配"——核心就是正则表达式。这一章把规则讲全。

grep 是什么:逐行筛选

grep 把输入一行一行地看,只要某行"包含"匹配的内容,就把整行打印出来。它可以读文件,也可以接在管道后面。

$ grep "error" app.log          # 从文件里筛出含 error 的行
$ ps aux | grep "nginx"        # 从上一个命令的输出里筛

grep 的常用参数

参数作用
-i忽略大小写(Error / ERROR / error 都匹配)
-v反选:打印"不匹配"的行(排除噪音常用)
-n显示行号
-r / -R递归搜整个目录(替代你"用 VSCode 全局搜")
-c只输出匹配的行数(count)
-o只打印匹配到的那部分,不是整行(配正则提取很有用)
-l只列出"哪些文件里有",不显示具体行
-w整词匹配(grep -w cat 不会匹配 category)
-A 3/-B 3/-C 3同时显示匹配行的后(After)/前(Before)/前后(Context)3 行
-E用扩展正则(ERE),少打很多反斜杠(强烈推荐)
 
$ grep -rn "TODO" src/             # 在 src/ 里递归找 TODO,带文件名和行号
$ grep -i "error" app.log | grep -v "timeout"  # 要 error 但排除 timeout
$ grep -C 2 "panic" app.log       # 出错那行的前后各 2 行,看上下文

正则表达式:到底怎么"匹配"

正则就是用一套"元字符"描述一类字符串的模式。先记住:普通字符匹配它自己,而下面这些字符有特殊含义。

元字符含义例子 → 匹配
.任意一个字符a.c → abc、axc、a9c
*前一个东西重复 0 次或多次ab*c → ac、abc、abbbc
+前一个重复 1 次或多次(需 -E)ab+c → abc、abbc(不含 ac)
?前一个出现 0 或 1 次(需 -E)colou?r → color、colour
^行首锚点^ERROR → 以 ERROR 开头的行
$行尾锚点;$ → 以分号结尾的行
[abc]方括号内任意一个[Gg]rep → Grep、grep
[^abc]方括号内取反(非 a/b/c)[^0-9] → 任意非数字
[a-z]范围[a-zA-Z0-9] → 字母数字
\转义,让元字符变回普通字符3\.14 → 字面的 3.14
{2,4}前一个重复 2 到 4 次(需 -E)[0-9]{3} → 恰好 3 位数字
(ab)+分组(需 -E)(ab)+ → ab、abab
a|b或(需 -E)cat|dog → cat 或 dog
⚠️ BRE vs ERE:为什么有时要加反斜杠

grep 默认用基础正则(BRE),里面 + ? { } ( ) | 是普通字符, 想用它们的"特殊含义"得加反斜杠:\+ \? \{ \} \( \) \|。
加上 -E 切到扩展正则(ERE)后,这些直接生效、不用反斜杠。 建议:凡是用到 + ? | () {} 就加 -E,省心很多。

对比:grep -E "ab+c" 等价于 grep "ab\+c"。

字符类(POSIX,方便又可读)

写法等价于
[[:digit:]][0-9] 数字
[[:alpha:]][a-zA-Z] 字母
[[:alnum:]]字母 + 数字
[[:space:]]空白(空格、Tab 等)
[[:upper:]]/[[:lower:]]大写 / 小写字母

"这个模式匹配什么"——读这张表练眼力

命令找的是
grep -E "^[0-9]+$"整行只有数字的行(^开头 $结尾夹住)
grep -E "^\s*#"注释行(开头可有空白,然后是 #)
grep -E "^$"空行
grep -vE "^\s*(#|$)"去掉注释行和空行(看配置文件神器)
grep -oE "[0-9]{1,3}(\.[0-9]{1,3}){3}"从文本里提取所有 IP 地址
grep -oE "\w+@\w+\.\w+"粗略提取邮箱
grep -E "error|fail|panic"含任一关键词的行
💡 提取而不是整行:-o + -E

默认 grep 给你整行。加 -o 就只吐出"匹配到的片段",特别适合从一大段文本里抠出 IP、邮箱、版本号。 常接着 | sort | uniq -c | sort -rn 做统计(第 14 章会用到)。

05文本三剑客与切分工具

grep 负责"筛行",而 cut/sort/uniq/sed/awk 负责"切列、排序、去重、替换、算字段"。组合起来威力巨大。

小工具:cut / sort / uniq / wc / tr

命令作用例子
cut按分隔符切列。-d 定分隔符、-f 选第几列cut -d: -f1 /etc/passwd → 所有用户名
sort排序。-n 按数字、-r 倒序、-h 按 4K/2M、-k2 按第 2 列、-u 排序兼去重sort -rn nums.txt
uniq去重相邻重复行(所以前面常先 sort)。-c 计数、-d 只看重复的sort x | uniq -c
wc统计。-l 行数、-w 词数、-c 字节wc -l file
tr逐字符替换/删除。常用大小写转换、压缩空白tr 'a-z' 'A-Z'
⚠️ uniq 只去"相邻"重复

uniq 不会全局去重,只合并挨在一起的相同行。所以几乎总是写成 sort | uniq, 或直接 sort -u。"统计每项出现几次"的黄金搭配是 sort | uniq -c | sort -rn。

sed —— 流式编辑(最常用:替换)

sed 逐行处理文本,最高频用途是"查找替换",语法 s/旧/新/(substitute)。

$ sed 's/foo/bar/' file        # 每行第一个 foo → bar(结果打屏,不改原文件)
$ sed 's/foo/bar/g' file       # g = 全局,一行里所有 foo 都换
$ sed -i 's/foo/bar/g' file    # -i = 直接改原文件(in-place)
$ sed -i.bak 's/foo/bar/g' file # 改之前先备份成 file.bak,更稳
$ sed '/^#/d' file            # 删除所有注释行(/模式/d)
$ sed -n '10,20p' file        # 只打印第 10~20 行(-n 静默 + p 打印)
💡 分隔符可换

替换路径时斜杠会打架,可把分隔符换成别的字符:sed 's#/old/path#/new/path#g',清爽很多。 sed 默认用 BRE,想用 + ? 等加 -E。

awk —— 按"列/字段"处理的利器

awk 自动把每行按空白切成字段:$1 第一列、$2 第二列……$0 整行,NF 字段总数,NR 行号。

$ awk '{print $1}' access.log         # 打印每行第 1 列(如访问 IP)
$ awk '{print $1, $NF}' file          # 第 1 列和最后一列($NF)
$ awk -F: '{print $1}' /etc/passwd   # -F 指定分隔符为冒号
$ awk '$3 > 100' data.txt             # 只输出第 3 列 > 100 的行(条件过滤)
$ awk '/error/ {print $2}' app.log     # 含 error 的行,打印第 2 列
$ awk '{sum += $1} END {print sum}' n.txt # 把第 1 列累加求和
📖 什么时候用 cut,什么时候用 awk?

分隔符固定、只想取某几列 → cut 更简单。但当列之间是多个空格/Tab 混合 (如 ps、ls -l 的输出),cut 会数错,awk 更稳——它默认把连续空白当一个分隔符。 需要条件判断、求和、用最后一列时也选 awk。

06find 实战 ★

你说习惯用 VSCode 连服务器搜文件——其实 find 在终端里更快、更强,还能"找到后直接批量处理"。这章按场景讲。

基本结构

find+ 从哪找(路径)+ 条件(名字/类型/时间…)+ 对结果做什么 
$ find . -name "*.log"     # 在当前目录(.)递归找所有 .log 文件
$ find /var/log -name "*.gz"  # 从 /var/log 开始找
⚠️ 通配符一定要加引号

find . -name *.log 不加引号,shell 会先把 *.log 展开成当前目录的文件名再传给 find,行为出错。 永远写成 -name "*.log",让 find 自己去匹配每一层。

常用条件

条件含义
-name "*.py"按名字(区分大小写),支持通配符
-iname "*.PY"同上但忽略大小写
-type f / -type d只要文件(f)/ 只要目录(d)
-mtime -7修改时间在 7 天内(+7 表示 7 天前,7 表示恰好第 7 天)
-mmin -3030 分钟内改过
-newer 文件比某个文件更新
-size +100M大于 100M(-size -1k = 小于 1k)
-empty空文件或空目录
-path "*/node_modules/*" -prune排除某些目录,不进去找
-maxdepth 2最多往下找 2 层(放在条件最前面)
 
# 找最近一天改过的 .py 文件
$ find . -name "*.py" -mtime -1

# 找大于 100M 的文件(清理磁盘前先看看谁在占空间)
$ find . -type f -size +100M

# 搜代码但跳过 node_modules 和 .git
$ find . -path "*/node_modules/*" -prune -o -name "*.js" -print

找到之后:做点什么(find 的精华)

找出来只是第一步,真正强大的是对每个结果执行命令。两种方式:

写法说明
-exec cmd {} \;{} 代表每个找到的文件,逐个执行(\; 结尾,注意空格)
-exec cmd {} +把多个文件一次性拼给命令(更快)
-deletefind 自带的删除,比 -exec rm 更安全(不怕空格)
... -print0 | xargs -0 cmd交给 xargs 批量处理,对超多文件更高效
 
# 在所有 .conf 文件里找含 "listen" 的行(find 替代 VSCode 全局搜内容!)
$ find /etc -name "*.conf" -exec grep -l "listen" {} +

# 删除 7 天前的临时日志(先把 -delete 换成 -print 预览,确认无误再删!)
$ find /tmp -name "*.log" -mtime +7 -print     # 先看
$ find /tmp -name "*.log" -mtime +7 -delete    # 再删

# 给所有 .sh 加可执行权限
$ find . -name "*.sh" -exec chmod +x {} +
💡 find vs grep -r vs VSCode 搜索

按文件名/属性找文件 → find;按文件内容找 → grep -rn "关键词" .; 两者结合 → find … -exec grep …。一旦上手,你会发现在 SSH 里几秒就搞定,比开 VSCode 连接快得多。 (还有更快的现代替代:fd 找文件、rg/ripgrep 找内容,若服务器装了的话。)

⚠️ -exec rm 前永远先 -print

批量删除是高危操作。养成习惯:先用 -print(或把命令换成 echo/ls)看清楚匹配到哪些, 眼睛确认无误,再把动作换成 -delete 或 rm。多打一条命令,换一夜安稳觉。

07权限 · 进程 · 信号 · 磁盘

权限位读懂了不再发怵,杀进程知道该用哪种信号、何时才动 kill -9。

权限位:rwxr-xr-x 怎么读

ls -l 开头那 10 个字符:第 1 个是类型,后 9 个分三组,分别给属主 / 属组 / 其他人。

  d  rwx  r-x  r-x
  │   │    │    └─ 其他人(others):可读可执行,不可写
  │   │    └────── 同组用户(group):可读可执行
  │   └─────────── 属主(user):可读可写可执行
  └─────────────── 类型:d=目录 -=文件 l=链接
权限对文件对目录数字
r 读看内容列出里面有啥(ls)4
w 写改内容增删里面的文件2
x 执行运行它能进入该目录(cd)1

chmod:改权限的两种写法

数字法:每组把 r=4、w=2、x=1 相加。rwx=7、r-x=5、rw-=6、r--=4。

$ chmod 755 script.sh   # rwxr-xr-x 属主全权,其他人可读可执行(脚本/程序常用)
$ chmod 644 notes.txt   # rw-r--r-- 属主可改,别人只读(普通文件常用)
$ chmod 600 id_rsa      # rw------- 只有自己能读写(私钥必须这样)
# 符号法:谁(u/g/o/a)+加/减/设(+/-/=)+权限(rwx)
$ chmod +x run.sh       # 给所有人加执行权(最常用)
$ chmod go-w file      # 去掉同组和其他人的写权限
⚠️ SSH 私钥权限太松会被拒绝登录

如果 ~/.ssh/id_rsa 权限不是 600(别人也能读),ssh 会直接报 "permissions are too open" 拒绝使用。修复:chmod 600 ~/.ssh/id_rsa。

属主与属组:文件"归谁"——chown

权限位(rwx)只是规则,规则作用在谁身上,取决于这个文件"属于谁"。每个文件都有两个归属:

$ ls -l app.log
-rw-r--r--  1  alice  devops  1.2K Jun  7 10:20 app.log
                  │       └─ 属组(group):devops 组的成员,适用"组"那一段权限(r--)
                  └───────── 属主(owner/user):alice,适用"属主"那一段权限(rw-)
# 不是 alice、也不在 devops 组的人 → 走最后一段"其他人"(r--)
📖 权限判定的顺序

系统看你是谁:① 你是属主吗?是 → 用属主那 3 位,判定结束(哪怕组/其他更宽也不管)。 ② 否则你在属组里吗?是 → 用组那 3 位。③ 都不是 → 用"其他人"那 3 位。 所以"我明明在组里却不能写"可能是因为你恰好是属主,而属主那段没给 w。

chown:改归属

$ sudo chown alice file           # 只改属主 → alice
$ sudo chown alice:devops file    # 同时改属主和属组(冒号分隔)
$ sudo chown :devops file          # 只改属组(冒号开头);等价于 chgrp devops file
$ sudo chown -R alice:alice /var/www  # -R 递归:整个目录树都改归属(最常用)
$ sudo chown --reference=好文件 坏文件  # 照着某个文件的归属来设(对不齐时方便)
⚠️ chown 几乎总要 sudo

普通用户不能把自己的文件"送"给别人,也不能随便认领别人的文件——改归属基本是 root 的特权, 所以 chown 前面通常得加 sudo。(改属组若目标组是你所属的组,可不用 sudo。)

你最常撞上的几个场景

现象原因 & 解法
用 sudo 建的文件,之后自己改不了 / git 报权限错那文件属主成了 root。sudo chown -R $USER:$USER 目录 收回归属($USER 是你当前用户名)
scp / 解压 / clone 下来一堆文件,属主乱了统一归到自己名下:sudo chown -R $USER:$USER ./项目
部署网站,Nginx/PHP 读不到文件(403)把目录交给 Web 服务用户:sudo chown -R www-data:www-data /var/www/site
明明 chmod 777 了还 Permission denied权限够但归属/上层目录不对;先 ls -l 看属主是谁、ls -ld 父目录 看进得去吗
💡 chmod 改"能不能"、chown 改"归谁"

两个常一起用但别搞混:chmod 调的是 rwx 权限,chown 调的是属主/属组 归属。 遇到 "Permission denied",先 ls -l 看清楚——是权限位不够(用 chmod),还是文件不归你(用 sudo chown)。 另:umask 决定你新建文件的默认权限,知道有这回事即可,一般不用动。

看进程:ps / pgrep / top

$ ps aux                       # aux=所有用户(a)+显示用户名(u)+无终端的后台进程(x);列 USER PID %CPU %MEM ... COMMAND
$ ps aux | grep nginx           # 找 nginx 相关进程(拿到 PID)
$ pgrep -a nginx              # 更直接:直接列出 nginx 的 PID 和命令行
$ top                          # 实时看 CPU/内存占用(htop 更好看,按 q 退出)
📖 PID 是什么

每个运行中的进程有个唯一编号 PID。要操作一个进程(暂停、杀掉),几乎都先得知道它的 PID—— 用 ps|grep 或 pgrep 拿到。

信号与杀进程:kill / pkill / killall

"杀进程"本质是给它发一个信号。不同信号含义不同,选对了才优雅。

信号编号含义
SIGTERM15默认信号:礼貌地请它退出,程序可以先保存、清理再走
SIGINT2等于你按 Ctrl+C,中断
SIGKILL9强杀:系统直接终结,程序来不及清理。最后手段
SIGHUP1"挂断";很多服务收到它会重载配置而不退出
 
$ kill 12345              # 默认发 SIGTERM(15),优雅退出 —— 永远先试这个
$ kill -9 12345           # 发 SIGKILL,强杀。仅当上面那条没反应时才用
$ kill -HUP 12345         # 让服务重载配置(等价 kill -1)
$ pkill nginx             # 按名字杀(不用先查 PID),杀所有匹配的
$ pkill -f "python app.py"  # -f 匹配完整命令行,更精准
$ killall nginx           # 按"进程名精确"杀(和 pkill 类似)
⚠️ 不要张口就 kill -9

-9 不给程序任何收尾机会,可能导致数据没存盘、临时文件没清、锁没释放。 正确顺序:① kill PID(SIGTERM)→ ② 等几秒看是否退出 → ③ 还赖着不走,再 kill -9。 另外 pkill -f python 这种太宽的匹配会误杀,先用 pgrep -af python 确认范围。

作业控制:前台 / 后台 / 不挂断

操作作用
cmd &直接放后台运行
Ctrl+Z把前台命令暂停并丢到后台
bg / fg让暂停的任务在后台继续 / 调回前台
jobs看当前 shell 的后台任务
nohup cmd &退出登录后仍继续运行(忽略挂断信号)
disown把已在后台的任务从 shell"脱钩",关终端也不死

完整流程:放后台 → 查看 → 切回来

不用再开好几个窗口。下面这套在同一个终端里就能管多个任务,关键是每个后台任务有个编号 %1 %2,靠它精确切换。

# ① 启动时就放后台:命令末尾加 &
$ ./train.sh &
[1] 34521                  # [1]=任务编号(job 1),34521=进程 PID

# ② 已经在前台跑着、想丢后台:先按 Ctrl-Z 暂停,再 bg 让它在后台继续
$ ./build.sh
^Z                          # 你按了 Ctrl-Z,它被"暂停"了
[2]+ Stopped   ./build.sh
$ bg                         # 让刚暂停的任务在后台继续跑(bg=background)

# ③ 看现在有哪些后台任务、各自编号
$ jobs
[1]-  Running   ./train.sh &
[2]+  Running   ./build.sh &

# ④ 切回来:fg + 编号(这就是你问的"怎么切回来")
$ fg %1                      # 把 1 号任务调回前台(fg=foreground)。光 fg 不带编号=最近那个
📖 %1 这个编号怎么用

jobs 里方括号的数字就是任务编号。凡是要指定某个任务,都用 %编号: fg %2 切回 2 号、kill %1 杀掉 1 号、bg %2 让 2 号后台继续。 带 + 的是"最近一个"(fg 不写编号就是它),- 是上一个。

⚠️ 这套 job 编号只在"当前这个 shell"里有效

jobs/fg/%1 管的是当前终端会话里的任务。一旦你关掉这个终端/断开 SSH, 普通后台任务(只加了 & 的)通常会跟着结束。想"关了还活着",要么 nohup ./x & + disown, 要么——更推荐——用下面说的 tmux。另外 & 放后台的任务仍会往屏幕打输出, 会和你后续操作混在一起,建议顺手重定向:./train.sh > train.log 2>&1 &。

💡 长任务别用 nohup,用 tmux

nohup ... & 能让任务在断线后存活,但看不到输出、不好管理。而你说的"老开好几个窗口很麻烦"—— 这正是 第 9 章 tmux 的主场:一个 SSH 连接里就能分屏 + 多窗口,各跑各的任务、随时切换, 断线重连还能回到原来的画面。&/fg 适合临时挪一下,长期多任务请直接用 tmux。

磁盘与链接

$ df -h                  # 各分区总量/已用/剩余(-h 人类可读)。磁盘满了先看这个
$ du -sh *               # 当前目录下每一项各占多大(-s 汇总 -h 可读)
$ ln -s /opt/app/bin/tool ~/tool   # 软链接(快捷方式),最常用
📖 软链接 vs 硬链接(一句话)

软链接(ln -s)像快捷方式,存的是"目标路径",目标删了就失效,可跨分区、可链目录。 硬链接(ln 不加 -s)是同一份数据的另一个名字,日常用软链接就够了。

08远程传输与打包 ★

本机 ↔ 服务器、服务器 ↔ 服务器之间怎么传文件,从网上怎么下载——这章全讲清。

scp:基于 SSH 的复制(简单直接)

记法:scp 来源 目的地。远程位置写成 用户@主机:路径。在你本机执行。

# ① 本机 → 服务器(上传)
$ scp ./report.pdf  user@1.2.3.4:/home/user/

# ② 服务器 → 本机(下载),注意来源和目的地调换
$ scp user@1.2.3.4:/var/log/app.log  ./

# ③ 整个目录:加 -r
$ scp -r ./dist  user@1.2.3.4:/var/www/

# ④ 服务器 A → 服务器 B(本机当中转发起)
$ scp userA@hostA:/data/x.tar  userB@hostB:/backup/
⚠️ 端口是大写 -P!

如果服务器 SSH 不是默认 22 端口,scp 指定端口用大写 -P 2222, 而 ssh 命令里却是小写 -p 2222——这俩大小写相反,是经典踩坑点。 记法:scp 大写 P、ssh 小写 p。

rsync:增量同步(传大目录/反复同步首选)

rsync 只传"有变化的部分",中断后能续传,远胜 scp 重传整个目录。同步代码、备份数据常用。

$ rsync -avz ./project/  user@host:/srv/project/
#         │││                  -a 保留权限/时间等  -v 显示过程  -z 压缩传输
$ rsync -avz --progress big.iso  user@host:/data/   # 显示进度条
$ rsync -avz --delete ./site/  user@host:/var/www/    # 让目标和源完全一致
$ rsync -avz -e "ssh -p 2222" ./x  user@host:~/   # 非默认端口这样写
⚠️ 源路径末尾的 / 有讲究

rsync -a src/ dst/ → 把 src 里面的内容放进 dst。
rsync -a src dst/(无斜杠)→ 把 src 这个目录本身放进 dst(变成 dst/src)。
而 --delete 会删掉目标里源没有的文件,用前务必先加 --dry-run 预演一遍。

curl 与 wget:从网络下载

需求命令
下载并按原名保存curl -O http://x/a.zip 或 wget http://x/a.zip
下载另存为别的名字curl -o new.zip http://...
跟随跳转(301/302)curl -L http://...(-L 很常用)
断点续传wget -c http://...(大文件断网后接着下)
调 API:带 header / POSTcurl -H "Authorization: Bearer xxx" -d '{...}' url
只看响应头curl -I http://...
静默 + 显示错误curl -sS http://...(脚本里常用)
📖 curl 还是 wget?

下载文件、镜像整站 → wget(自带断点续传 -c、递归 -r 更顺手)。 调接口、调试 HTTP、当管道一环 → curl(默认把内容打到 stdout,方便接 | jq 处理 JSON,见第 12 章)。

打包压缩:tar / gzip / zip

tar 的参数像咒语,但记住两条就够 90% 场景。口诀:"创建看 c,解开看 x"。

$ tar -czf backup.tar.gz  ./project   # 打包+压缩:c 创建 z gzip f 文件名
$ tar -xzf backup.tar.gz             # 解开:把 c 换成 x(extract)
$ tar -tzf backup.tar.gz             # 只看里面有啥不解开:t(list)
$ tar -xzf x.tar.gz -C /opt/         # 解到指定目录:-C
字母含义
c / x / t创建 create / 解开 extract / 列表 list(三选一)
z用 gzip(对应 .tar.gz;.tar.bz2 用 j,.tar.xz 用 J)
f后面紧跟文件名(必须有,且通常放最后)
v显示处理过程(verbose,可选)

跨平台传给 Windows 用户,用 zip -r out.zip dir/ 打包、unzip out.zip 解开更通用。 单个文件压缩用 gzip file(变 file.gz)、gunzip file.gz 还原。

09会话保持与 Shell 效率

每天在服务器干活,这两样东西能极大省时间:tmux 让你断线不丢进程,history/补全让你少打字。

tmux:断线不丢、可分屏的"会话"

痛点:SSH 一断,跑着的程序就没了。tmux 把会话留在服务器端,你重连后能原样接回。

$ tmux new -s work     # -s 给会话起个名(session),这里叫 work;在里面随便跑你的长任务
# —— 此时网断了 / 你手动按 Ctrl-b 然后 d 脱离(detach)——
$ tmux ls               # 重新登录后,看有哪些会话还活着
$ tmux attach -t work  # -t 指定要接回哪个会话(target),画面和任务都还在!

三层心智模型:会话 > 窗口 > 面板

tmux 的层次想清楚,后面的快捷键就不乱了。用浏览器类比很贴切:

tmux 概念类比说明
会话 session整个浏览器一个项目开一个,断线后留在服务器、能整体接回
窗口 window标签页 tab一个会话里多个窗口,底部状态栏一字排开,数字切换
面板 pane同屏分屏把一个窗口切成几块,同时看到(比如左边跑服务、右边看日志)
📖 这就是你"少开几个窗口"的答案

以前开 3 个 SSH 窗口 = 现在一个会话里 3 个 window(或几个 pane)。只连一次 SSH, 全在里面切,断线全保住。底部状态栏会显示当前在第几个窗口、叫什么名。

快捷键:先按前缀,松手再按下一个

tmux 所有快捷键都先按前缀键 Ctrl+b,松开,再按功能键。下面 C-b 代表这个前缀。

10SSH 进阶

天天 ssh 登录,配好这几样能少打很多字、少输很多密码,还能打通内网服务。

~/.ssh/config:给主机起别名

不想每次都打一长串 ssh -p 2222 user@1.2.3.4?在本机 ~/.ssh/config 里写一次:

# ~/.ssh/config
Host prod
    HostName 1.2.3.4
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_rsa

之后直接 ssh prod 就登录了,scp file prod:~/ 也认这个别名。

ssh-copy-id:免密登录

原理一句话:把公钥放到服务器,服务器用它验证你手里的私钥,匹配就放行,不再问密码。

$ ssh-keygen -t ed25519        # -t 指定密钥类型(type),ed25519 是现在推荐的算法;一路回车即可
$ ssh-copy-id prod              # 把公钥装到服务器(这一次要输密码)
$ ssh prod                      # 以后再登录,免密!
⚠️ 私钥不外传、权限收紧

id_rsa(无 .pub)是私钥,绝不能发给别人或上传。id_rsa.pub 才是公钥,可以公开。 私钥权限须 chmod 600。

端口转发(隧道):访问"够不着"的服务

场景:服务器上跑着只监听 localhost:8080 的服务(数据库、面板、Jupyter),你本机直接访问不到。 用本地转发,把它"映射"到你本机端口:

$ ssh -L 9000:localhost:8080  prod
# 之后浏览器开 http://localhost:9000,流量经 SSH 隧道直达服务器的 8080
选项方向用途
-L 本地:目标:端口本地 → 远程本机访问服务器(或它能到的)内网服务
-R 远程:目标:端口远程 → 本地把本机服务暴露给服务器那边

跳板机:经堡垒机连内网

很多公司要先登堡垒机(jump host)才能到内网机器。ProxyJump 一步到位:

$ ssh -J user@jump  user@10.0.0.5     # 经 jump 跳到内网 10.0.0.5
# 也可写进 ~/.ssh/config:
Host inner
    HostName 10.0.0.5
    ProxyJump user@jump

11服务 · 日志 · 定时任务

运维日常:启停服务、看服务日志、设定时任务。现代 Linux 用 systemd 这一套。

systemctl:管理服务

$ systemctl status nginx       # 看状态:在跑吗?最近几行日志
$ sudo systemctl restart nginx # 重启(改完配置后)
$ sudo systemctl stop nginx    # 停止 / start 启动 / reload 重载配置
$ sudo systemctl enable nginx  # 设为开机自启(disable 取消)
$ systemctl is-active nginx    # 脚本里判断是否在运行
📖 enable ≠ start

start 是现在启动一次;enable 是设置开机时自动启动。两者独立—— 想"现在就启动且以后开机也启动",用 systemctl enable --now nginx。

journalctl:看服务日志

$ journalctl -u nginx           # 只看 nginx 这个服务的日志
$ journalctl -u nginx -f        # -f 实时跟踪(和 tail -f 一样,排障时开着)
$ journalctl -u nginx --since "10 min ago"   # 只看最近 10 分钟
$ journalctl -u nginx -p err -b     # 本次开机后(-b)的错误级别(-p err)日志

传统的日志文件仍在 /var/log/ 下(如 /var/log/syslog),配合 tail -f 和 grep 看。

crontab:定时任务

crontab -e 编辑当前用户的定时任务,crontab -l 列出。每行 5 个时间字段 + 命令:

# ┌── 分(0-59)
# │ ┌── 时(0-23)
# │ │ ┌── 日(1-31)
# │ │ │ ┌── 月(1-12)
# │ │ │ │ ┌── 周几(0-7,0和7都是周日)
# * * * * *  要执行的命令
  30 3 * * *   /home/user/backup.sh        # 每天 03:30
  */10 * * * * curl -s http://localhost/ping # 每 10 分钟
  0 9 * * 1   /home/user/weekly.sh        # 每周一 09:00
⚠️ cron 里环境和你登录时不一样

cron 的 PATH 很精简,务必用绝对路径写命令和脚本,别依赖 ~ 或自定义变量。 排查时把输出重定向到文件:... >> /tmp/cron.log 2>&1,这样才看得到为什么没跑成功。

12排查类工具(出问题时救命)

"端口被谁占了""磁盘怎么满了""这段 JSON 怎么读"——这些工具关键时刻顶用。

谁占了端口 / 谁在听:ss、lsof

$ ss -tlnp                  # 列出所有在监听的 TCP 端口及对应进程
#    │││└ p=显示进程  n=不解析成域名(更快)  l=只看 listen  t=TCP
$ ss -tlnp | grep :8080     # 8080 被谁占着
$ lsof -i :8080             # 同样查 8080,lsof 更直观(可能需 sudo)
$ lsof /var/log/app.log     # 哪个进程正打开着这个文件(删不掉文件时查它)
💡 "Address already in use" 怎么办

启动服务报端口被占,流程:ss -tlnp | grep :端口 找到 PID → 确认那进程是不是该杀 → kill PID(别急着 -9)。netstat 是更老的同类工具,新系统用 ss。

jq:处理 JSON(配合 curl)

API 返回一坨压扁的 JSON,jq 帮你格式化、提取字段。

$ curl -s https://api.github.com/repos/torvalds/linux | jq .
# . = 原样美化输出(缩进+高亮)
$ curl -s .../repo | jq .stargazers_count   # 取某个字段
$ cat data.json | jq '.items[] | .name'     # 遍历数组取每项的 name
$ cat data.json | jq '.items | length'      # 数组长度

watch:周期性重复一条命令

$ watch -n 2 'df -h'          # 每 2 秒刷新一次磁盘用量
$ watch -d 'ss -tlnp | grep :80'  # -d 高亮每次的变化部分

磁盘满了:定位大文件/大目录

$ df -h                       # ① 先看哪个分区满了
$ du -sh /var/* | sort -rh | head   # ② 逐层往下找谁最大(详见第 14 章配方)
$ ncdu /var                    # ③ 有 ncdu 的话,交互式浏览最省事
📖 du 和 df 对不上?

有时 df 显示满了,但 du 加起来没那么多——常因为有进程还占着已删除的文件 (空间没真正释放)。用 lsof | grep deleted 找到那个进程,重启它即可回收。

13echo · 变量 · 环境 ★

你说见过 echo "..." >> 某文件 但没完全理解。其实它是三个零件拼起来的,拆开就清楚了。

零件一:echo —— 把文字输出到 stdout

$ echo "hello"           # 屏幕打印 hello(其实是写到了 stdout)
hello
$ echo -n "no newline"   # -n 结尾不换行
$ echo -e "a\tb\nc"      # -e 解释转义:\t 制表 \n 换行

关键:echo 的输出默认去屏幕,但因为它走的是 stdout,所以能被第 3 章的 >/>> 改道到文件。

零件二:$ —— 变量展开

shell 看到 $名字,会在运行前把它替换成变量的值。系统预置了一批环境变量:

$ echo $HOME            /home/user     # 你的家目录
$ echo $USER            user          # 当前用户名
$ echo $PWD             /home/user/x   # 当前目录
$ echo "嗨 $USER,家在 ${HOME}"          # ${} 是更明确的写法
嗨 user,家在 /home/user
⚠️ 单引号 vs 双引号 —— 最关键的区别

双引号 ":变量会展开。单引号 ':原样输出,$ 不展开。

$ echo "我在 $HOME"     我在 /home/user   # 双引号:展开了
$ echo '我在 $HOME'      我在 $HOME       # 单引号:原样,没展开

记法:要里面的变量"活"→ 双引号;要"一字不改"(比如写正则、写带 $ 的字面量)→ 单引号。

自定义变量,以及 export 的意义

$ NAME="world"            # 定义变量。注意:等号两边绝不能有空格!
$ echo "hi $NAME"          hi world
$ export NAME="world"     # export = 让子进程/脚本也能看到这个变量
⚠️ NAME = "x" 会报错

赋值等号两侧不能有空格:NAME="x" ✅,NAME = "x" ❌ (shell 会把它当成"运行命令 NAME,参数是 = 和 x")。这是初学最常踩的坑。

📖 export 到底干嘛?(shell 变量 vs 环境变量)

不加 export 的变量只在当前 shell 有效;你运行的程序、脚本(子进程)看不到它。 export 把它"升级"成环境变量,子进程才会继承。所以配置程序行为(如 export API_KEY=...)要用 export。

PATH:命令为什么"找得到"

你敲 git,shell 怎么知道去哪找这个程序?它按顺序在 $PATH 列的目录里挨个找。

$ echo $PATH
/usr/local/bin:/usr/bin:/bin:/home/user/.local/bin   # 冒号分隔的目录列表
$ which git              # 看 git 到底用的是哪个路径
/usr/bin/git

"命令找不到 command not found"通常就是程序所在目录不在 PATH 里。把目录加进去:

$ export PATH="$HOME/bin:$PATH"   # 把 ~/bin 加到最前面(本次会话有效)

压轴:逐字拆解 echo "..." >> ~/.bashrc

现在把三个零件拼起来。这是往配置文件追加一行的标准写法:

$ echo 'export PATH="$HOME/bin:$PATH"' >> ~/.bashrc
echo '...'  → 产生一行文本(单引号:保留字面的 $HOME,不在此刻展开) ↓ >>  → 把这行追加到文件末尾(用 > 会清空整个 .bashrc,灾难!) ↓ ~/.bashrc  → 你的 shell 启动配置文件,以后每次开终端都会读它 
$ source ~/.bashrc      # 改完让它立刻生效(否则要重开终端)。也可写作 . ~/.bashrc
⚠️ 为什么这里偏要用单引号?

我们想往文件里写的是字面字符串 $HOME/bin:$PATH,让它以后每次启动时再展开。 若用双引号,$HOME/$PATH 会在写入的此刻就被替换成当时的值,写进去的就成了死值。所以这里用单引号。

alias 与配置文件

$ alias ll='ls -lah'     # 给长命令起短别名;写进 ~/.bashrc 永久生效
$ alias gs='git status'
📖 .bashrc vs .bash_profile

粗略记:每开一个新终端都会读 .bashrc(别名、函数、PATH 放这); .bash_profile 主要在登录时读一次。大多数发行版会让 .bash_profile 去 source .bashrc, 所以日常配置统一往 .bashrc 写就好。

14组合命令实战配方 ★

前面学的命令,真正的威力在于用管道串起来解决具体问题。每条配方都拆开讲每一段在干嘛。

📖 一个通用范式

大多数"一行流"都长这样:
生成列表 | 过滤(grep) | 变换(awk/cut) | 排序(sort) | 取头部(head) | 批量执行(xargs)。 看懂这个骨架,再长的命令也能拆着读。

① 找出最大的 5 个文件 → 删除

先看清楚,再删。第一步只看,不删:

$ du -ah . | sort -rh | head -n 5
1.2G  ./data/dump.sql
800M  ./logs/app.log
...
这一段在干嘛
du -ah .列出当前目录下每个文件(-a)的大小,人类可读(-h)
sort -rh按大小倒序排(-r),且能正确比较 1.2G > 800M(-h 识别单位)
head -n 5只取前 5 行

确认无误后再删。用 find 按大小筛 + -delete 最安全:

$ find . -type f -size +500M -print     # 先看:大于 500M 的文件有哪些
$ find . -type f -size +500M -delete    # 眼睛确认后再删
⚠️ 别用 rm $(...) 直接套娃

诱人但危险的写法 rm $(du -ah | sort -rh | head -5 | ...) 有两个雷: ① 文件名带空格会被拆成多个参数,删错东西;② 看不到要删什么就执行了。 要批量删,优先 find ... -delete,或 find ... -print0 | xargs -0 rm(用 NUL 分隔防空格)。 永远先 -print 预览。

② 日志里出现次数最多的 IP / URL(Top N)

$ awk '{print $1}' access.log | sort | uniq -c | sort -rn | head
  1532 203.0.113.7
   841 198.51.100.2
awk '{print $1}'取每行第 1 列(访问 IP)
sort排序,让相同 IP 挨在一起(uniq 的前提)
uniq -c合并相同行并在前面计数
sort -rn按这个计数数字倒序排
head取前 10(Top 10)

这个 sort | uniq -c | sort -rn 三连是"统计频率"的万能套路,把 $1 换成 URL 列、状态码列即可复用。

③ 批量重命名 / 批量改后缀

# 把所有 .txt 改成 .md(用 shell 循环,清晰可控)
$ for f in *.txt; do mv "$f" "${f%.txt}.md"; done
# ${f%.txt} = 去掉结尾的 .txt;注意 "$f" 一定加双引号防空格

# 找出并批量处理(文件名可能含空格的安全姿势)
$ find . -name "*.jpeg" -print0 | xargs -0 -I{} mv {} {}.bak
# -I{} = 用 {} 当占位符代表"每个文件名",这样能把文件名插到命令中间(而不只是接在末尾)

④ 其他常用一行流

目的命令
统计项目代码总行数find . -name "*.py" | xargs wc -l | tail -1
最近 1 天改过的文件find . -type f -mtime -1
各子目录占多大、排序du -sh */ | sort -rh
实时看日志里的错误tail -f app.log | grep --line-buffered -i error
--line-buffered:逐行立即输出,管道里实时刷新不卡缓冲
杀掉某程序的所有进程pgrep -f myapp | xargs kill
去重并计数某列cut -d, -f3 data.csv | sort | uniq -c | sort -rn
看某进程实时资源watch -n1 'ps -p $(pgrep -f myapp) -o %cpu,%mem'
ps -p 指定 PID、-o 只输出指定列(这里要 CPU 和内存)
💡 拆长命令的方法

遇到看不懂的一行流,用 | 把它切成几段,从左往右一段段跑:先只跑第一段看输出, 再加第二段……每一步都能看到中间结果,逻辑立刻清楚。这也是自己调试管道的标准方法。

15Shell 脚本基础 ★

你会 Python,编程概念跳过。这章只讲 bash 跟你直觉不一样的那些点和坑。

开头三件套:shebang + 可执行 + 健壮模式

#!/usr/bin/env bash          # shebang:告诉系统用 bash 跑它
set -euo pipefail            # 健壮模式(强烈建议,下面解释)

echo "hello"
$ chmod +x script.sh    # 加可执行权限
$ ./script.sh             # 运行(或 bash script.sh 直接跑,不必 chmod)
💡 set -euo pipefail 是什么,为什么开头就写

默认 bash 很"宽容",出错也继续往下跑,很危险。这行让脚本变严格:

  • -e 任何命令出错(返回非 0)就立刻退出,不带病继续。
  • -u 用到未定义变量就报错(防 rm -rf $DIR/ 里 DIR 是空导致删根目录的惨剧)。
  • -o pipefail 管道里任一环失败,整条算失败(否则只看最后一个命令)。

把它当成 Python 里"该抛异常就抛异常",而不是默默吞掉。

变量:和 Python 几个明显不同

name="world"              # ① 等号两边不能有空格!name = "x" 会报错
echo "hi $name"            # ② 用变量要加 $;字符串拼接直接挨着写
count=$(ls | wc -l)         # ③ $(...) 命令替换:把命令输出存进变量
total=$((3 + 5))           # ④ $((...)) 才是算术;a+b 不会自动算数
⚠️ 变量永远加双引号:"$var"

这是 bash 第一大坑。不加引号,变量值里的空格会被拆成多个参数,空值会"消失"。

f="my file.txt"
rm $f      # ❌ 变成 rm my file.txt → 试图删 "my" 和 "file.txt" 两个
rm "$f"    # ✅ 正确:当作一个文件名

规则简单:除非你明确想要分词,否则一律 "$var"。

脚本参数与退出码

变量含义
$1 $2 …第 1、2…个参数
$0脚本自身的名字
$#参数个数
"$@"所有参数(各自独立,带引号最安全)
$?上一条命令的退出码:0=成功,非 0=失败
$$当前脚本的 PID

条件判断:[[ ... ]]

bash 的"if 后面跟的是命令",依命令退出码判断真假(0 为真)。测试条件用 [[ ]]:

if [[ "$count" -gt 0 ]]; then
    echo "有文件"
elif [[ -z "$name" ]]; then      # -z:字符串为空
    echo "name 是空的"
else

set -euo pipefail

log() {                          # 定义函数,参数同样用 $1 $2
    echo "[$(date +%H:%M:%S)] $1"
}

target="${1:-/tmp}"            # 取第 1 个参数,没传则默认 /tmp

if [[ ! -d "$target" ]]; then
    log "目录不存在: $target"
    exit 1                       # 以非 0 退出码表示失败
fi

📖 函数"返回值"的约定

16Vim 系统教学 ★

Vim 不是"难",是它有一套和别的编辑器完全不同的心智模型。理解了模型,快捷键就不是死记了。

📖 vi 和 vim 是什么关系？

vi 是 Unix 最早的纯文本编辑器(1976 年诞生),所有 Linux/Unix 系统都自带。vim 全称 Vi IMproved(vi 改进版),在 vi 基础上加了语法高亮、多级撤销、可视模式、插件等现代功能。 你在服务器上敲 vi 和 vim 可能打开同一个程序——很多发行版(如 Ubuntu)的 vi 命令实际上是 vim 的兼容模式。

实际区别:

  • 极简容器/嵌入式系统(如 Docker 的 alpine 镜像、busybox)可能只装了精简版 vi,没有 vim。打开后没有语法高亮、u 只能撤销一次、不支持 Ctrl+v 列块模式。
  • 如果你在服务器上敲 vim 提示 command not found——那就用 vi 顶一顶,功能差不多,快捷键大部分通用。也可以装: apt install vim / yum install vim。
  • vi 兼容模式下,方向键可能失效(只能 hjkl)、退格键可能变 ^?。这些都是 vi 的「老毛病」,vim 全修了。

本教程讲的是 vim,但绝大部分内容(移动、编辑、搜索替换、:wq)在 vi 里完全一样。如果你用的真是远古 vi,遇到不支持的功能(如列块 Ctrl+v)不要慌,换 vim 就行。

⚠️ 常见坑：输入后要按 Esc 才能生效

💡 c vs d 怎么选

18压缩与归档 tar/zip ★

把一堆文件打成包、压成小体积——备份、分发、传输前的必修课。这章把 tar / gzip / zip 一次讲透。

概念先行：归档 vs 压缩

这是两个独立动作，只是经常一起做：

动作干什么典型工具
归档 archive把多个文件/目录捆成一个文件，保留目录结构和元信息（权限、时间戳）tar
压缩 compress把数据用算法压小（去冗余），省磁盘/省传输时间gzip / bzip2 / xz

tar 本身只管打包不管压缩——.tar 文件和里面内容一样大。配上压缩标志才是常见的 .tar.gz / .tar.bz2 / .tar.xz。

tar 基本功：创建 / 解开 / 查看

tar 的参数看起来像咒语，但记住这条铁律就够了：「c 创建，x 解开，t 查看，f 跟文件名」。

# === 创建 ===
$ tar -cf project.tar  ./project          # 只打包不压缩
$ tar -czf project.tar.gz  ./project       # 打包 + gzip 压缩（最常用）
$ tar -cjf project.tar.bz2 ./project       # 打包 + bzip2 压缩（更小但更慢）
$ tar -cJf project.tar.xz  ./project       # 打包 + xz 压缩（最小最慢）

# === 解开 ===
$ tar -xf project.tar                      # 解开 tar
$ tar -xzf project.tar.gz                  # 解开 .tar.gz
$ tar -xjf project.tar.bz2                 # 解开 .tar.bz2
$ tar -xJf project.tar.xz                   # 解开 .tar.xz

# === 只看不解 ===
$ tar -tzf project.tar.gz                  # 列出里面有什么（不解开）
$ tar -tzf project.tar.gz | grep .py      # 只看 .py 文件在不在包里
标志含义记忆口诀
ccreate 创建想「造」包用 c
xextract 解开想「拆」包用 x
tlist 列出内容想「偷看」用 t
ffile 指定文件名必须要有，且放最后（f 后面紧跟文件名）
vverbose 啰嗦模式显示正在处理的文件名，可选但推荐
zgzip 压缩/解压配 .tar.gz
jbzip2 压缩/解压配 .tar.bz2
Jxz 压缩/解压配 .tar.xz（大写！）
C指定解压到的目录-C /opt/app（大写！）
⚠️ 经典踩坑

① f 必须在最后，后面紧跟文件名——tar -zcfv x.tar.gz dir/ 是错的（v 挡在 f 和文件名之间）。正确：tar -czvf x.tar.gz dir/。
② 解压会覆盖同名文件，没有确认提示。不确定内容时先用 -t 看看。
③ 现代 tar(GNU tar)其实能自动检测压缩格式，写 tar -xf x.tar.gz 不写 z 也 OK——但建议还是写全，清晰且兼容老系统。

tar 进阶用法

# 解压到指定目录（-C 大写）
$ tar -xzf app.tar.gz -C /opt/app/

# 只解压包里某几个文件
$ tar -xzf backup.tar.gz ./project/config.yml ./project/data/

# 打包时排除某些文件/目录
$ tar -czf src.tar.gz --exclude=node_modules --exclude=*.log ./project

# 打包时排除列表文件
$ tar -czf src.tar.gz --exclude-from=.tarignore ./project  # .tarignore 每行一个排除模式

# 不解压直接看某个文件内容
$ tar -xzf backup.tar.gz -O ./project/readme.txt  # -O 输出到 stdout

# 从管道接收 tar（常见于远程传输场景）
$ ssh user@host "tar -czf - /data" | tar -xzf -C ./backup   # 远程打包→本地解
💡 管道传输妙用

上面最后一条命令的行末 - 表示 stdin/stdout——远端打包写到 stdout，管道传到本地，本地从 stdin 解开。整个过程不产生中间文件，大目录传输时非常高效。更多用法见第 8 章 scp/rsync。

gzip / bzip2 / xz：单独压缩单个文件

如果不打 tar 包，只想压一个文件（比如日志），直接用压缩工具：

# gzip — 最常用，速度快
$ gzip app.log                        # → app.log.gz（原文件消失）
$ gzip -k app.log                      # → app.log.gz（-k 保留原文件）
$ gunzip app.log.gz                    # 解压
$ zcat app.log.gz                      # 不解压直接看内容
$ zless app.log.gz                     # 不解压分页浏览

# 调节压缩级别：1 最快但压缩比低，9 最慢但最狠（默认 6）
$ gzip -1 huge.log                      # 快速压缩大日志
$ gzip -9 important.tar                 # 极致压缩
工具后缀压缩比速度适用场景
gzip.gz中快日常首选；日志压缩；管道友好
bzip2.bz2较高较慢对体积敏感、不赶时间的归档
xz.xz最高最慢长期归档、发布包（很多发行版用 .tar.xz）

zip / unzip：跨平台的通用格式

传给 Windows/macOS 用户时用 zip——无需额外软件就能打开。

# 打包+压缩
$ zip -r project.zip ./project            # -r 递归目录，必须加！
$ zip -r -q project.zip ./project          # -q 安静模式，不打印文件列表
$ zip -r -e secret.zip ./docs             # -e 加密，会提示输入密码

# 解压
$ unzip project.zip                     # 解压到当前目录
$ unzip project.zip -d /opt/app/        # 解压到指定目录（-d，注意和 tar 的 -C 不同！）
$ unzip -l project.zip                   # 只看里面有什么不解压（-l = list）
$ unzip -o project.zip                   # 覆盖已有文件不询问（overwrite）
⚠️ zip 与 tar 的差异

① zip 的递归标志是 -r（和 cp/scp 一样），tar 则是默认递归不需要额外标志。
② zip 的指定目标目录是 -d，tar 是 -C——容易搞混。
③ Linux 文件权限/属主信息在 zip 里会丢失，tar 则完整保留。跨平台分发给最终用户用 zip，服务器间传输/备份用 tar.gz。

Git 不只是「存代码」——它是你改崩了能回滚、多条线并行开发不打架、多人协作不互相踩脚的保障。这章从日常高频操作讲到 worktree 这种效率利器。

Git 记录的历史

Git 跟踪的是修改（delta），不是文件。每个 commit 是一次完整快照，但你看到的 diff 是相邻两次快照的差。理解这个模型，后续所有操作都有直觉。

起步：安装与配置

# 安装（一般已预装）
$ git --version                       # 确认已装

# 必须配置的两项（提交时会附上你的身份）
$ git config --global user.name "Junlin Wu"
$ git config --global user.email "junlin@example.com"

# 推荐顺手配的
$ git config --global init.defaultBranch main  # 默认分支名用 main
$ git config --global pull.rebase true        # pull 时用 rebase 代替 merge
$ git config --global core.editor vim           # 默认编辑器设为 vim

从零开始：init 与 clone

# 方式一：把现有目录变成 Git 仓库
$ cd my-project/
$ git init                              # 创建 .git/ 子目录，这就是仓库本体
$ git add -A                            # 把所有文件加入暂存
$ git commit -m "初始提交"             # 第一次 commit

# 方式二：从远程克隆（最常见）
$ git clone git@github.com:user/repo.git
$ git clone git@github.com:user/repo.git  my-dir   # 克隆到指定目录
$ git clone --depth 1 git@...repo.git             # 浅克隆：只拉最新 commit

日常四步：status → add → commit → push

90% 的日常工作就是这四个命令的循环：

# ① 看当前状态——改了什么、哪些待暂存、哪些待提交
$ git status
$ git status -s                       # 精简输出

# ② 把修改加入暂存区
$ git add README.md                    # 加一个文件
$ git add -A                            # 加所有修改（新增+修改+删除）
$ git add -u                            # 只加已跟踪文件的修改和删除
$ git add -p                            # 交互式逐块暂存（推荐！精细控制）

# ③ 提交——给这次修改起个名字
$ git commit -m "修复登录页按钮不显示的问题"
$ git commit -am "小修"               # 跳过 add 直接提交已跟踪文件的修改

# ④ 推送到远程
$ git push                             # 推到默认远程（通常是 origin）
$ git push -u origin main              # 首次推送并设置上游
💡 好的 commit message

用祈使句（像对代码库下命令）："添加用户登录功能" 而不是 "加了登录"。 第一行 50 字以内概括做了什么，空一行后可写详细的为什么。这样 git log --oneline 一目了然。

看历史：log / diff / show

# log — 查看提交历史
$ git log                              # 完整日志（q 退出）
$ git log --oneline                   # 一行一条，最常用
$ git log --oneline --graph --all    # 带 ASCII 分支图，看合并历史
$ git log --oneline -10               # 只看最近 10 条
$ git log --oneline main..feature      # feature 有但 main 没有的提交
$ git log -p app.js                    # 看某文件的变更历史（含具体 diff）

# diff — 看具体改了什么
$ git diff                             # 工作区 vs 暂存区（还没 add 的改动）
$ git diff --staged                   # 暂存区 vs 最新 commit（已 add 还没 commit 的）
$ git diff HEAD~1                     # 工作区 vs 上一次 commit
$ git diff main feature               # 两个分支的差异

# show — 看某次 commit 的详细信息
$ git show                            # 最新 commit 的完整 diff
$ git show a1b2c3d                     # 指定 commit 的 diff
$ git show a1b2c3d:README.md           # 看某 commit 时某个文件的内容

分支：并行开发的基石

分支就是一条独立的开发线。不改主线代码，在自己的分支上随意折腾，好了再合回去。

# 查看分支
$ git branch                           # 列出本地分支（前面有 * 的是当前分支）
$ git branch -a                        # 含远程分支
$ git branch -v                        # 显示各分支最新 commit

# 创建与切换
$ git branch feature-login             # 创建分支（但不切过去）
$ git switch feature-login             # 切换分支（Git 2.23+ 推荐用 switch）
$ git switch -c feature-payment       # 创建并切换（一步到位）← 最常用
$ git checkout -b feature-payment      # 等价写法（老习惯）

# 合并
$ git switch main                      # 先切回目标分支
$ git merge feature-login              # 把 feature-login 合到 main

# 删除分支
$ git branch -d feature-login          # 安全删除（已合并的才能删）
$ git branch -D feature-login          # 强制删除（即使没合并）
📖 switch vs checkout

老版本 Git 用 git checkout 既切换分支又恢复文件，职责太重容易误操作。 Git 2.23 起拆成两个命令：git switch（切分支）和 git restore（恢复文件）。 如果你用的是 Git ≥ 2.23（大概率是），建议养成用 switch/restore 的习惯。

rebase：让历史更干净

merge 会产生一条「合并提交」的岔路；rebase 则是把 feature 分支的提交「搬」到 main 最新 commit 后面，历史是一条直线。

# feature 分支 rebase 到 main
$ git switch feature
$ git rebase main                     # 把 feature 的提交搬到 main 最新处
# 如有冲突：解决冲突 → git add → git rebase --continue

# 交互式 rebase：整理自己的提交历史（神器）
$ git rebase -i HEAD~3               # 整理最近 3 个 commit
# 在编辑器中：pick 保留 / squash 合并到上一个 / reword 改消息 / drop 丢弃
⚠️ rebase 黄金法则

永远不要 rebase 已经推送给别人使用的分支。 rebase 会改写 commit hash，别人本地的旧 commit 和远程的「新」commit 会产生冲突地狱。 简单说：自己的私有 feature 分支随便 rebase，共享分支（main/develop）只用 merge。

远程操作：fetch / pull / push / remote

# 查看远程仓库
$ git remote -v                       # 列出远程地址
$ git remote add upstream git@.../original.git  # 添加上游仓库（fork 场景）

# fetch：只拉取不合并（安全！先看看远程有什么变化）
$ git fetch origin
$ git fetch --prune                   # 同时清理本地已不存在的远程分支引用

# pull = fetch + merge（或 fetch + rebase）
$ git pull                            # 拉取并合并
$ git pull --rebase                  # 拉取并 rebase（保持历史直线）

# push：把本地提交推到远程
$ git push origin main
$ git push --force-with-lease         # 强制推送（比 --force 安全，会检查远程是否有新提交）
$ git push origin --delete old-branch  # 删除远程分支

撤销与回滚：从轻到重

场景命令说明
改乱了工作区，想丢弃git restore 文件名回到最近一次 add/commit 的状态（不可逆！）
已 add 但想撤回暂存git restore --staged 文件名文件不改，只从暂存区移除
改 commit messagegit commit --amend -m "新消息"替换最新 commit（未 push 时用）
漏了文件想补进上次 commitgit add 漏的文件 && git commit --amend --no-edit不改消息，只补文件
撤销最近 N 个 commitgit reset --soft HEAD~3commit 撤销，改动回到暂存区
彻底丢弃最近 N 个 commitgit reset --hard HEAD~3⚠️ 改动全部消失，慎用
撤销某次 commit（已 push）git revert a1b2c3d产生一个新 commit 来"反向"撤销，不改历史
⚠️ reset --hard 是不可逆的

如果 commit 过，还有救：用 git reflog 找到丢失的 commit hash，再 git reset --hard <hash> 回来。 如果没 commit 过就丢了……那就真没了。

git add 后悔了怎么撤销

这是最高频的「手滑」场景——git add . 一把梭，加进去才发现不该加（临时文件、调试代码、机密信息）。别慌，好几种救法：

# ① 撤回某个文件的暂存（文件改动还在，只是从暂存区移除）
$ git restore --staged config.yml       # 新写法（Git 2.23+，推荐）
$ git reset HEAD config.yml              # 老写法（所有版本通用）

# ② 撤回所有暂存（整个暂存区清空，改动全回到工作区）
$ git restore --staged .                 # 新写法
$ git reset HEAD .                       # 老写法
$ git reset                              # 等价于 git reset HEAD .（最简）

# ③ 不仅撤回暂存，连文件改动也丢弃（回到最近一次 commit 的状态）
$ git restore config.yml                 # 丢弃工作区改动（不可逆！）
$ git checkout -- config.yml            # 老写法

# ④ 已 commit 但没 push：撤回最近一次 commit，改动回到暂存区
$ git reset --soft HEAD~1                 # commit 消失，改动在暂存区
$ git reset HEAD~1                        # 默认 --mixed：commit 消失，改动回到工作区（不在暂存）

简单记：只撤暂存不改文件 → git restore --staged <文件>；连文件改动一起丢 → git restore <文件>；连 commit 一起撤 → git reset --soft HEAD~1。

stash：临时「存档」手头改动

正在改 feature 分支，突然要切去 main 修紧急 bug——但手头代码还不适合 commit。stash 就是「暂存现场，回头继续」。

$ git stash                            # 暂存所有改动，工作区变干净
$ git stash push -m "WIP: 登录页重构一半"  # 带说明的暂存
$ git stash list                       # 看所有暂存
$ git stash pop                        # 恢复最近一次暂存并删掉记录（最常用）
$ git stash apply stash@{1}            # 恢复指定暂存但不删除
$ git stash drop stash@{1}             # 删掉指定暂存

⭐ git worktree：同时开多个分支

这是 Git 最被低估的功能之一。worktree 让你在同一个仓库里同时 checkout 多个分支到不同目录。 再也不需要为了切分支而 stash → switch → 改完 → switch → stash pop 了。

📖 什么场景用 worktree？

① 正在 A 分支写代码，突然要切 B 分支修 bug——用 worktree 两个分支同时开着，互不干扰。
② 长时间运行的程序/编译在 A 分支跑着，想在 B 分支干别的。
③ 对比两个分支的代码——直接开两个终端窗口各看一个目录。
④ CI/CD 中在不同目录 checkout 不同版本来做并行测试。

# 查看所有 worktree
$ git worktree list
/home/me/project          a1b2c3d [main]

# 添加新 worktree：在 ../project-hotfix 目录 checkout hotfix 分支
$ git worktree add ../project-hotfix hotfix
# 如果 hotfix 分支不存在，加 -b 创建：
$ git worktree add -b hotfix ../project-hotfix main

# 现在两个目录各有一个分支，各自独立操作：
$ cd ../project-hotfix                # 进入 hotfix 的 worktree
$ vim fix.py                           # 改代码
$ git add -A && git commit -m "修复"
$ git push
$ cd ../project                        # 回到原来的 worktree，一切如初

# 用完了删掉 worktree
$ git worktree remove ../project-hotfix
# 或手动删目录后清理引用：git worktree prune
⚠️ worktree 注意事项

① 同一个分支不能在两个 worktree 同时 checkout——Git 会拒绝。
② 每个 worktree 有自己的 stash 和 暂存区，但共享同一个 .git（实际是引用回主仓库），所以 commit 全部共享。
③ git worktree remove 比手动 rm -rf 干净——它会清理 .git/worktrees/ 里的元数据。
④ worktree 目录最好放在仓库外——别在已有 worktree 里再嵌套创建。

💡 实战技巧：用 worktree 做 code review

别人提了 PR，你想在本地看他的代码但不污染自己的工作区：

$ git fetch origin pull/42/head:pr-42  # 拉 PR 到本地分支
$ git worktree add ../review-pr42 pr-42 # 在新 worktree 打开
# review 完
$ git worktree remove ../review-pr42
$ git branch -D pr-42                   # 清理分支
